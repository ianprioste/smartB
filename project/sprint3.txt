═════════════════════════════════════════════════════════════════════════════
🎯 SKU GENERATION RULES
═════════════════════════════════════════════════════════════════════════════

1. PARENT_PRINTED (Produto Principal Estampado)
   Format: {MODEL_CODE}{PRINT_CODE}
   Example: CAMSTPV
   ├─ CAM = Model code
   ├─ STPV = Print code
   └─ Dependencies: None (parent of all variations)

2. VARIATION_PRINTED (Variação Estampada)
   Format: {MODEL_CODE}{PRINT_CODE}{COLOR_CODE}{SIZE}
   Example: CAMSTPVBRP
   ├─ CAM = Model code
   ├─ STPV = Print code
   ├─ BR = Color code
   ├─ P = Size
   └─ Dependencies: [PARENT_PRINTED, BASE_PLAIN]

3. BASE_PLAIN (Base Lisa) - REFERENCED & AUTO-SEEDED (NEW in 3.3/3.4)
   Format: {MODEL_CODE}{COLOR_CODE}{SIZE}
   Example: CAMBRP
   ├─ CAM = Model code
   ├─ BR = Color code
   ├─ P = Size
   ├─ Must exist before creating variations
   ├─ NEW (3.3): Auto-seeded if missing (referenced but not in Bling)
   ├─ NEW (3.4): Safe creation even without template
   └─ NEW (3.4): Marked with AUTO_SEED reason when auto-created


═════════════════════════════════════════════════════════════════════════════
✅ VALIDATION RULES & SAFEGUARDS
═════════════════════════════════════════════════════════════════════════════

1. Template Existence
   ├─ Each model must have BASE_PLAIN template configured
   ├─ Each model must have PARENT_PRINTED template configured
   ├─ BLOCKED if template missing (3.0-3.2)
   └─ NEW (3.4): Safe fallback for None template_payload - can still create base

2. Size Validation
   ├─ Selected sizes must be subset of model.allowed_sizes
   ├─ NEW (3.4): Centralized via _get_model_sizes() helper
   ├─ At least one size must be selected
   └─ Error if invalid size provided

3. Model/Color Configuration
   ├─ Models must exist in config
   ├─ Models must be active (is_active=true)
   ├─ Colors must exist in config
   └─ Colors must be active

4. Dependency Chain
   ├─ PARENT_PRINTED has no dependencies
   ├─ VARIATION_PRINTED depends on [PARENT_PRINTED, BASE_PLAIN]
   ├─ BLOCKED if dependency missing
   └─ NEW (3.3): Auto-seed BASE_PLAIN if missing

5. Product Existence (Bling)
   ├─ Query Bling for existing products by SKU
   ├─ CREATE if not exists
   ├─ UPDATE if exists and needs change
   ├─ NOOP if exists and correct
   ├─ Read-only queries (no writes)
   ├─ NEW (3.3): Caching to prevent rate-limit issues
   └─ NEW (3.4): Safe fallback when Bling query fails

6. Auto-Seed Safeguards (NEW in 3.4)
   ├─ Only auto-seed if not already in Bling
   ├─ Only for BASE_PLAIN type
   ├─ Only if dependency chain requires it
   ├─ Never auto-seed if template payload is None (safe fallback)
   └─ Mark with AUTO_SEED reason for transparency

💡 LESSONS LEARNED
═════════════════════════════════════════════════════════════════════════════

1. Dry-Run First
   └─ Preview before execution reduces errors dramatically
   └─ Users can review and cancel if something is wrong

2. Domain Logic Isolation
   └─ Pure functions in app/domain/ are easy to test
   └─ No mocking needed for business rules

3. Progressive Disclosure
   └─ Wizard guides user step-by-step
   └─ Reduces cognitive load

4. User Feedback Matters
   └─ User rejected manual mode → automatic only
   └─ User wanted Portuguese labels → implemented everywhere
   └─ User wanted template replacement → upsert pattern
   └─ User wanted clear auto-seed indication → AUTO_SEED notification

5. Status Messages
   └─ Loading modal with progressive messages keeps user informed
   └─ Reduces anxiety during long operations

6. Fail Fast
   └─ Don't retry expired refresh tokens
   └─ Show error immediately and guide to reauth

7. Code Quality Matters
   └─ Helper methods eliminate duplication and bugs
   └─ Safe fallbacks prevent crashes
   └─ Cleaner code = easier maintenance

8. Legacy Cleanup is Important
   └─ Remove dead code to reduce confusion
   └─ Docker/npm based deployment replaces shell scripts
   └─ Archived documentation still available in git history

═════════════════════════════════════════════════════════════════════════════
🎯 SKU GENERATION RULES
═════════════════════════════════════════════════════════════════════════════

1. PARENT_PRINTED (Produto Principal Estampado)
   Format: {MODEL_CODE}{PRINT_CODE}
   Example: CAMSTPV
   ├─ CAM = Model code
   ├─ STPV = Print code
   └─ Dependencies: None (parent of all variations)

2. VARIATION_PRINTED (Variação Estampada)
   Format: {MODEL_CODE}{PRINT_CODE}{COLOR_CODE}{SIZE}
   Example: CAMSTPVBRP
   ├─ CAM = Model code
   ├─ STPV = Print code
   ├─ BR = Color code
   ├─ P = Size
   └─ Dependencies: [PARENT_PRINTED, BASE_PLAIN]

3. BASE_PLAIN (Base Lisa) - REFERENCED ONLY
   Format: {MODEL_CODE}{COLOR_CODE}{SIZE}
   Example: CAMBRP
   ├─ CAM = Model code
   ├─ BR = Color code
   ├─ P = Size
   └─ Must exist before creating variations
   
NOTE: STAMP (Estampa) was REMOVED per user request
      No longer generated in plans

✅ VALIDATION RULES
═════════════════════════════════════════════════════════════════════════════

1. Template Existence
   ├─ Each model must have BASE_PLAIN template configured
   ├─ Each model must have PARENT_PRINTED template configured
   └─ BLOCKED if template missing

2. Size Validation
   ├─ Selected sizes must be subset of model.allowed_sizes
   ├─ At least one size must be selected
   └─ Error if invalid size provided

3. Model/Color Configuration
   ├─ Models must exist in config
   ├─ Models must be active (is_active=true)
   ├─ Colors must exist in config
   └─ Colors must be active

4. Dependency Chain
   ├─ PARENT_PRINTED has no dependencies
   ├─ VARIATION_PRINTED depends on [PARENT_PRINTED, BASE_PLAIN]
   └─ BLOCKED if dependency missing

5. Product Existence (Bling)
   ├─ Query Bling for existing products by SKU
   ├─ CREATE if not exists
   ├─ UPDATE if exists and needs change
   ├─ NOOP if exists and correct
   └─ Read-only queries (no writes)

   └─ Fail-fast on expired refresh token
   └─ Easy reauth with modal + redirect

💡 LESSONS LEARNED
═════════════════════════════════════════════════════════════════════════════

1. Dry-Run First
   └─ Preview before execution reduces errors dramatically
   └─ Users can review and cancel if something is wrong

2. Domain Logic Isolation
   └─ Pure functions in app/domain/ are easy to test
   └─ No mocking needed for business rules

3. Progressive Disclosure
   └─ Wizard guides user step-by-step
   └─ Reduces cognitive load

4. User Feedback Matters
   └─ User rejected manual mode → automatic only
   └─ User wanted Portuguese labels → implemented everywhere
   └─ User wanted template replacement → upsert pattern

5. Status Messages
   └─ Loading modal with progressive messages keeps user informed
   └─ Reduces anxiety during long operations

6. Fail Fast
   └─ Don't retry expired refresh tokens
   └─ Show error immediately and guide to reauth
